# Практическая работа 3: Настройки запуска и базовая защита веб службы

## Запуск приложения

### Учебный режим (по умолчанию)
```bash
cd node
npm start
```

### Боевой режим
```bash
cd node
npm start -- --mode=боевой
```

или через переменную окружения:
```bash
cd node
set APP_MODE=боевой
npm start
```

или изменив файл конфигурации `config/appsettings.json`:
```json
{
  "app": {
    "mode": "боевой",
    ...
  }
}
```

## Источники настроек и приоритет

Порядок приоритета (от низкого к высокому):
1. Файл `config/appsettings.json`
2. Переменные окружения (`APP_MODE`, `APP_PORT`, `APP_TRUSTED_ORIGINS`, `APP_READ_PER_MINUTE`, `APP_WRITE_PER_MINUTE`)
3. Аргументы командной строки (`--mode=`, `--port=`, `--trustedOrigins=`, `--readPerMinute=`, `--writePerMinute=`)

## Критичные настройки и их обоснование

### 1. Режим работы (mode)
- **Критичность**: Высокая
- **Почему**: Учебный режим предоставляет подробные сообщения об ошибках (полезно для отладки), боевой — минимальные (не раскрывает внутреннюю информацию злоумышленнику)

### 2. Доверенные источники (trustedOrigins)
- **Критичность**: Очень высокая
- **Почему**: CORS ограничивает доступ к API только с разрешённых доменов. Пустой список полностью блокирует работу. Проверка URL при запуске предотвращает ошибки конфигурации (опечатки, неправильные схемы)

### 3. Лимиты частоты (rateLimits)
- **Критичность**: Высокая
- **Почему**: Защита от DDoS и злоупотреблений API. Разные лимиты для чтения и записи: чтение безопаснее (ниже риск), запись требует более строгих ограничений

### 4. Порт (port)
- **Критичность**: Средняя
- **Почему**: Должен быть в допустимом диапазоне (1-65535). Невалидный порт заблокирует запуск

### 5. Защитные заголовки
- **X-Content-Type-Options**: Предотвращает MIME-sniffing атаки
- **X-Frame-Options**: Защита от clickjacking (встраивание в iframe)
- **Cache-Control**: Предотвращает кэширование чувствительных данных
- **Pragma**: Дополнительная защита от кэширования для HTTP/1.0

## Раннее обнаружение ошибок конфигурации

Реализовано в `src/config.js`:
1. Все настройки проверяются ДО запуска сервера
2. При ошибках приложение завершается с кодом 1
3. Выводятся понятные сообщения о том, что не так
4. Это предотвращает:
   - Запуск с неправильной конфигурацией
   - Утечку информации через неправильные настройки безопасности
   - Непредсказуемое поведение в production

## Режимы работы

| Параметр | Учебный | Боевой |
|----------|---------|--------|
| Сообщения об ошибках | Подробные | Минимальные |
